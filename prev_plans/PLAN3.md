# Chess Engine Hackathon — Plan 3: "Chess Commander"

## The Idea: Natural Language Chess

Chess is fundamentally a game of high-level strategy (the "fun" part) bogged down by grueling tactical calculation and opening memorization (the "miserable" part). Existing platforms like chess.com suffer from high bounce rates because beginners and intermediates hate blundering a piece to a basic tactic and losing instantly.

**"Chess Commander"** completely shifts the paradigm of how chess is played. 

You no longer drag and drop pieces. You do not need to memorize the Sicilian Defense. Instead, you play as a General sitting in the commander's tent, giving high-level, natural language orders to an AI army.

> **User types:** *"Lock down the center, prepare a slow queenside expansion, and try to trade off our dark-squared bishop."*
> **Engine:** Translates the vibes into dynamically weighted evaluation parameters, runs a lightning-fast search, and plays `c4`.
> **Claude:** *"Understood, Commander. Starting queenside expansion with c4. The center remains solid."*

This hits all four hackathon criteria:
- **Chess quality**: Rust search engine is highly performant and handles the grueling tactical calculations flawlessly.
- **AI usage**: Claude acts as an essential interpreter, bridging messy human natural language strategy with explicit algorithmic engine weights. 
- **Process**: Two highly decoupled streams (Rust engine parameterization vs. LLM interpretation layer).
- **Engineering**: Real-time websocket streaming, prompt engineering, and dynamic engine parameter evaluation.

## The Translation Pipeline

The secret sauce of this hackathon project is the `llm_translator.py`. When a user submits a strategy, Claude is provided with the current FEN and asked to output a strictly formatted JSON of engine weights.

**Example Claude Interpretations:**
* *"I want to launch a devastating attack on the enemy king, I don't care about material."* -> `{"king_safety_weight": 5.0, "material_weight": 0.5, "mobility_weight": 2.0}`
* *"Let's play super passively, defend everything, build a fortress."* -> `{"pawn_structure_weight": 3.0, "center_control_weight": 0.5, "king_safety_weight": 4.0}`
* *"Try to trade down, we are up a pawn."* -> `{"simplification_bonus": 2.0}`

The Rust engine's `eval.rs` reads these parameters via extended UCI commands (e.g. `setoption name EvalWeights value {...}`) before running `go movetime 1000`.

## User Flow

1. **The Tactical Map:** The user lands on the web interface. A chessboard is visible, but the pieces are locked. Next to it is a military-style dispatch interface (chatbox).
2. **Opening Order:** The user types: *"Claim the center aggressively and develop the knights quickly."*
3. **Transmission:** A loading state shows: *Translating orders to battalion...*
4. **Execution:** Claude converts the order into a JSON weight config, passes it to the Rust engine, and the engine plays `e4`. 
5. **Debrief:** The chat feeds back: *"The infantry has moved to e4, preparing the center for the cavalry. Awaiting next orders."* (Generated by Claude post-move).
6. **The Opponent:** The opponent (either a standard Stockfish AI, or another human Commander) takes their turn. The user watches the board state change.
7. **Refining Strategy:** As the game progresses, the user must adapt their orders. *"They are attacking our queenside! Retreat the bishops and lock the pawn structure."*

## Two Parallel Streams

### Stream A — The Dynamic Rust Engine (2 people)

**Phase 1 (hours 1–4)**:
- Build the standalone Rust UCI engine using `shakmaty`.
- Implement standard move generation, Alpha-Beta pruning, and Quiescence search.
- **Key Divergence:** Instead of hardcoded evaluation weights, refactor evaluation constants into a `Config` struct that can be updated at runtime.

**Phase 2 (hours 5–8)**:
- Implement custom UCI commands to accept new evaluation configurations.
- Ensure changing evaluation parameters does not break Transposition Table lookups (or flush the TT upon parameter change).
- Focus on search speed; since the AI handles human errors, the engine needs to be as tactically bulletproof as possible to avoid blundering the grand strategy.

### Stream B — The General's Tent (2 people)

**Phase 1 (hours 1–4)**:
- Write the `llm_translator.py` and document `prompts/strategy_interpreter.md`.
- Build an automated test suite sending 20 different natural language string variations to Claude, enforcing strict JSON schema responses.
- Build the FastAPI backend and Websocket scaffolding.

**Phase 2 (hours 5–8)**:
- Integrate the React frontend.
- Implement the "Debrief" engine, where Claude takes the Engine's move (e.g., `Nf3`) and translates it back into a flavorful military report based on the user's initial prompt.
- Connect the full pipeline: UI -> FastAPI -> Claude (interpretation) -> Rust (evaluation & move) -> Claude (debrief) -> UI.

## Testing Strategy

| Test | Tool | Purpose |
|------|------|---------|
| Translation Schema | Pytest | Ensures Claude strictly returns the JSON format required by the Rust engine. |
| Tactical Baseline | pytest-chess | Send standard positions to the Rust engine; ensure basic tactical soundness remains. |
| Extreme Weight Constraints | Custom Script | Test the Rust engine with absurdly skewed evaluation tables (e.g., Queen = 1 pawn) to ensure it doesn't crash or hang. |
| End-to-End Latency | Manual | Ensure the entire pipeline (User -> Claude -> Rust -> Claude -> UI) completes in <3 seconds. |

## Stretch Goals

- **"Emergency Intercept":** Allow the user to "override" an order if they spot a forced mate they want to execute manually.
- **Visual Intelligence:** Highlight sections of the board in red/green (heatmaps) corresponding to the engine's current dynamic evaluation weights.
- **Multiplayer Commander Mode:** Host a matchmaking server so two humans can battle each other purely using natural language strategies.
